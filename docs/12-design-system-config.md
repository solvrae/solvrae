# 12 — Design System & Configuration

> **Status: partially implemented.** The shared-theme architecture below is real —
> generated repos ship one `packages/ui-theme` with a fixed **neutral** token set
> that every family consumes. The *configurable prompt* (base color, radius, dark
> mode, icon library, React-only `--base` / `--preset`) is **designed but not yet
> implemented**; today the theme is the neutral default. shadcn's own prompts never
> appear regardless. This doc describes the target design.

This document answers a common question: **do shadcn's native CLI prompts (base
color, style, base primitive, etc.) still appear when using Solvrae?**

**No — by design.** Solvrae owns the design-system decisions, asks them **once**
in its own unified prompt, normalizes them, and writes them into every
`packages/ui-<family>/components.json` itself. shadcn's interactive prompts never
surface, because:

1. Solvrae runs shadcn operations **non-interactively** (or writes config
   directly) — consistent with the "program against contracts" principle in
   [08 — Compatibility & Maintainability](08-compatibility-and-maintainability.md).
2. Each UI family's CLI (`shadcn`, `shadcn-vue`, `shadcn-svelte`) asks **different
   questions with different options**. Letting them through would re-introduce the
   exact fragmentation Solvrae exists to remove.
3. `solvrae add component` **never re-prompts** — it reads the stored
   `components.json`, so every component added later automatically matches the
   chosen design system.

## shadcn CLI v4 reality (March 2026)

The configuration surface changed in shadcn CLI v4. The classic `default` vs
`new-york` "style" model gave way to:

- **`--base`** — choose the underlying primitive: **Radix** or **Base UI**.
- **`--preset <code>`** — a *design-system preset*: a short code (created via
  `shadcn/create`) packing **colors, theme, icon library, fonts, radius**, shareable
  across teams and switchable mid-project.
- **`--monorepo`**, plus inspection flags **`--dry-run`, `--diff`, `--view`**.

> **Named styles are real and delivered via presets.** `shadcn/create` ships named
> design-system styles — **Nova, Vega, Maia, Lyra, Mira, Luma, Sera, Rhea** (e.g.
> [Sera](https://ui.shadcn.com/docs/changelog/2026-04-sera), April 2026, a
> typography-first style, available for both Radix and Base UI). Picking a style +
> base color + theme + fonts in `shadcn/create` produces a **preset code**
> (`--preset b0…`) — the style is *encoded in the preset*, not a fixed `style` enum.
> So Solvrae deliberately does **not** hardcode a style list; it exposes
> **`--preset <code>`** pass-through (React family). New styles shadcn adds later
> require **zero changes** to Solvrae.

Crucially, `--base`, presets, and "style" are **shadcn/ui (React) specific**.
`shadcn-vue` and `shadcn-svelte` have **no identical** equivalents. So not every
option maps across families.

## Solvrae's normalized design-token model

Solvrae defines a small, family-neutral model asked once at `init`, applied to
**all** families, then a set of **React-only extras** applied to `ui-react` only.

### Common subset (all families) → written to the one shared `ui-theme/styles.css`

| Option | Maps to | Default |
|--------|---------|---------|
| Style / preset | token set baked into `ui-theme/styles.css` | **Nova** |
| Base color | `@theme` color tokens in `ui-theme/styles.css` | `neutral` |
| Radius | CSS `--radius` token in `ui-theme/styles.css` | `0.5rem` |
| Dark mode | theme strategy (class-based) | enabled |
| Icon library | `lucide` (react/svelte) / `lucide-vue-next` (vue) | per-family default of `lucide` |

Because the token names are identical across React/Vue/Svelte, these live in a
**single shared file** (`packages/ui-theme/styles.css`) that every family and app
imports — see [05 — Generated Output](05-generated-output.md). Change them once,
all families update.

> **Default style = Nova.** If the user picks nothing, Solvrae bakes the **Nova**
> token set into `ui-theme/styles.css` as a built-in (no network/`--preset` code
> required). Override with `--preset <code>` (from `shadcn/create`) or a curated
> `--style <name>` convenience.

### React-only extras (applied to `ui-react` only)

| Option | Notes |
|--------|-------|
| Base primitive | `--base radix` \| `base` (Radix vs Base UI) |
| Preset / Style | `--preset <code>` pass-through. Named styles (Nova, Sera, …) are presets generated by `shadcn/create` — paste the code; no hardcoded style enum |

For Vue/Svelte families these are silently omitted; their `components.json` uses
the common subset plus that registry's own sane defaults. This is the
"apply per-family" decision — maximize what each registry offers without forcing a
lowest-common-denominator UX.

## The init prompt (illustrative)

```
◆  Base color?       › Neutral / Zinc / Stone / Slate / Gray
◆  Border radius?    › 0   0.3rem   0.5rem (default)   0.75rem   1rem
◆  Dark mode?        › Yes (class-based) / No
◆  Icon library?     › Lucide (default) / Radix Icons
─ React detected ──────────────────────────────────────────
◆  Style / preset?   › Nova (default) / Vega / Sera / … / Paste --preset code
◆  Base primitive?   › Radix (default) / Base UI
```

In `--yes` / CI mode every prompt falls back to the documented default. All values
are also settable via flags:

```bash
npx create-solvrae my-app -t next \
  --base-color zinc --radius 0.75rem --dark \
  --icon lucide --base radix --preset <code> --yes
```

## Where it's stored & how it propagates

- **Tokens** (style/preset, base color, radius, dark mode) → the single
  `packages/ui-theme/styles.css`. Each family's `components.json` `tailwind.css`
  points at this shared file.
- **Per-family settings** (icon library, React-only base/preset) → that family's
  `components.json`.
- `solvrae add template <id>` for a **new family** just adds a component-only
  package that imports the existing `ui-theme` — so a later `ui-vue` inherits the
  exact base color, radius, and dark-mode strategy already in place, with no
  re-prompting and nothing to reconcile.
- `solvrae add component` reads `components.json`; components inherit the design
  system, and any token additions land in the shared `ui-theme` (deduped).
- Because tokens are centralized, there is **no cross-family token drift** to fix —
  the class of bug `doctor` used to reconcile largely disappears.

## Changing the design system later

- Edit `packages/ui-theme/styles.css` directly — one file, all families update.
  Solvrae respects hand-edits (wiring is diff-based).
- Or a future `solvrae config set` / `solvrae upgrade` to re-apply a new base color
  or `--preset` to the shared theme with a diff preview before writing.
